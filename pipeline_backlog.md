# Maket CP — Pipeline: концепция и бэклог

> Сводный рабочий документ по модулю «Пайплайн» в Maket CP.
> Часть 1 — чеклист всех зафиксированных решений (со ссылками на `pipeline_spec.html`).
> Часть 2 — бэклог по блокам с приоритетами, зависимостями и критериями готовности.
>
> Поддерживается вместе с:
> - `pipeline_spec.html` — подробная спецификация (авторитетный источник).
> - `pipeline_mockup.html` — визуальный макет трёх метрик и ветвления.
> - `backlog.html` — общий бэклог проекта со статусами и тестами.

Дата последнего обновления: 2026-04-14. (Частичный apдейт статусов 2026-04-23 — см. changelog внизу.)

## Часть 1. Чеклист зафиксированных концепций

### 1.1. Зона охвата и позиционирование

- [x] **Три фазы продакшна: pre / prod / post.** Сейчас закрываем postproduction целиком, расширение в prod через шаблоны, созданные заранее, и real-time заполнение на площадке. → `pipeline_spec.html#scope`
- [x] **Core message.** Помогаем команде понять клиента, помогаем клиенту получить ожидаемый результат. → memory `project_core_message.md`
- [x] **Client validation frame.** Хаотичному клиенту продукт даёт структуру; организованному — подтверждает, что процесс идёт по плану. → memory `project_client_validation.md`
- [x] **Монетизация.** Per-project (~500₽) + студийная подписка, хранение 3 месяца, всегда доступен экспорт. → `pipeline_spec.html#roadmap-storage`, memory `project_monetization.md`

### 1.2. Модель пайплайна

- [x] **Два пайплайна:** общий (проект) + пер-фото (`_stage` у каждого фото). → `pipeline_spec.html#overview`
- [x] **8 этапов** (preselection, selection, cc, cc_approval, retouch, retouch_approval, tech, final). → `pipeline_spec.html#stages`
- [x] **17 триггеров** переходов, каждый с actor/channel. → `pipeline_spec.html#triggers`
- [x] **Этапы появляются динамически** (комментарии ретуши — только если аннотации есть; ЦК — только если загружена версия CC). → `pipeline_spec.html#stage-list`
- [x] **Ветвление:** карточки и фото движутся независимо. Несколько итераций CC/Retouch = несколько версий. → `pipeline_spec.html#branching`

### 1.3. Версии фото

- [x] **Модель версий:** глобальный порядковый номер + атрибут (cc / retouch / tech) + статус. → `pipeline_spec.html#versions`
- [x] **Единый справочник атрибутов** (cc/retouch/tech — одни и те же строки во фронте, бэке, БД). → `pipeline_spec.html#data-single`
- [x] **Стандартный порядок атрибутов:** orig → CC → retouch → tech. → `pipeline_spec.html#roadmap-standard`
- [x] **Preview versions по этапам:** превью тянет лицо/версию, соответствующую текущему этапу карточки. → memory `project_preview_versions.md`
- [x] **Persistent photo ID через COS keyword** (живёт между этапами и системами). → memory `project_preview_versions.md`

### 1.4. Данные

- [x] **`photo_versions` (Supabase)** — таблица версий. Поля, RLS, индексы. → `pipeline_spec.html#data-version`
- [x] **`stage_events`** — подписанные события переходов (actor, channel, timestamp). → `pipeline_spec.html#data-events`
- [x] **Timestamps per state на фото:** entered_queue_at, entered_work_at, uploaded_at, approved_at и т.д. — основа аналитики и SLA. → `pipeline_spec.html#roadmap-timestamps`, memory `project_photo_timestamps.md`
- [x] **Storage RLS:** `auth.uid()` в политике, `SECURITY DEFINER` только для RLS-bypass лукапа, не читать JWT в plpgsql. → memory `feedback_supabase_storage_rls.md`

### 1.5. UI представления

- [x] **Compact** (экран «Съёмки») — три числа на этапе: **готово / на этапе / ожидается**. Числа cumulative (N прошли — M в отборе). → `pipeline_spec.html#ui-compact`, memory `project_pipeline_metrics.md`
- [x] **Detailed** (оверлей по клику) — батч-движения per-photo. → `pipeline_spec.html#ui-detailed`, memory `project_per_photo_pipeline.md`
- [x] **Universal «awaiting push» filter** — для всех ролей, показывает что ждёт моего решения. → `pipeline_spec.html#roles`, memory `project_signed_approval.md`

### 1.6. Согласование

- [x] **Согласование = подписанное событие.** Identity-recorded (actor + channel), не роль-gated. Любой авторизованный участник может пушнуть любой этап. → `pipeline_spec.html#roadmap-partial`, memory `project_signed_approval.md`
- [x] **Три кнопки в каждом меню согласования:** согласовать всё / согласовать частично / запросить изменения. Scope = границы своего меню. → `pipeline_spec.html#roadmap-partial`, memory `project_partial_approval.md`
- [x] **Частично согласованное не убивается** — остаётся на этапе ждать своего ордера. → memory `project_partial_approval.md`
- [x] **Зоны согласования не закрепляем заранее** — команда самоорганизуется через coordination (future planning module закроет). → memory `project_signed_approval.md`

### 1.7. Эволюция метрик (v1 → v2)

- [x] **v1 (сейчас):** одно ведро «на этапе». → `pipeline_spec.html#roadmap-metrics`
- [x] **v2:** «на этапе» разделится на 4 состояния — `в очереди / в работе / загружено / ждёт согласования` — как только появятся соответствующие триггеры. → `pipeline_spec.html#roadmap-metrics`, memory `project_metrics_evolution.md`

### 1.8. Мёртвые ветки

- [x] **Kill на любом этапе.** История сохраняется для подсчёта «бессмысленной» работы и ретро. → `pipeline_spec.html#roadmap-dead`, memory `project_dead_branches.md`

### 1.9. Undo / Redo

- [x] **Ctrl/Cmd+Z на всех экранах**, 20 действий в localStorage, сброс при навигации.
- [x] **Завершающие этап действия требуют подтверждения** — не попадают в Undo-стек. → `pipeline_spec.html#roadmap-undo`, memory `project_undo_redo.md`

### 1.10. Далёкое будущее

- [x] **Модуль планирования** («маленькая Jira для фотографов»): продюсер строит pipeline конструктором, сравнение «план vs факт» для ретро. → `pipeline_spec.html#roadmap-planning`, memory `project_planning_module.md`
- [x] **Расширение потенциала и масштаба.** → `pipeline_spec.html#roadmap-potential`
- [x] **Хранение и доставка файлов.** → `pipeline_spec.html#roadmap-storage`

### 1.11. Связанные модули (не-pipeline, но влияют)

- [x] Аннотации ретуши (tag levels, circle markers, COS sync). → memory `project_retouch_annotations.md`
- [x] Article tool (rename + AI Vision). → memory `project_article_tool.md`
- [x] Мобильный клиент (карусель + галерея). → memory `project_mobile_client.md`

## Часть 2. Бэклог работ по блокам

Приоритеты:
- **P0** — foundation, без этого ничего не работает.
- **P1** — MVP пайплайна, первая рабочая версия.
- **P2** — улучшение UX и аналитика.
- **P3** — далёкое будущее.

Статусы:
- `todo` — не начато.
- `wip` — в работе.
- `done` — готово и проверено.
- `blocked` — заблокировано (указать чем).

## Как пользоваться этим файлом

- **Редактировать:** менять статусы в таблицах, добавлять новые задачи в соответствующий блок. После каждого изменения — коммит.
- **Проверка синхронизации:** раз в спринт сверяться с `pipeline_spec.html` и `MEMORY.md` — если там появились новые концепции, отразить здесь.
- **Приоритизация:** P0 задачи блокируют всё остальное. Не начинать P1 пока не закрыты P0 соответствующего блока.
- **Связь с общим `backlog.html`:** крупные задачи (на уровне эпиков: «ввести _stage на фото», «загрузка версий», «undo») должны параллельно появиться в `backlog.html` с отдельными подзадачами под тесты Maша.
