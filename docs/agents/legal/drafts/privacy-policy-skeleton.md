# Политика конфиденциальности Maket CP — skeleton

**Статус:** draft-скелет, не публиковать без review живого юриста.
**Версия:** v0.1, 2026-04-23.
**Автор:** Legal Counsel (LC).
**Применимое право:** 152-ФЗ РФ (основа) + GDPR (для международных клиентов, как best practice) + белорусский закон №99-З (базовая совместимость).
**Формат публикации:** отдельная страница на лендинге `/privacy` или `/privacy.html`. Обновление существующего `privacy.html` (последний апдейт 2026-04-20).

## Структура (10 разделов)

### 1. Вводная часть

- Кто оператор: ИП [ФИО], ИНН [XXXXXXXX], адрес для корреспонденции.
- Контакт для вопросов о ПДн: email [{contact_email}] + форма обратной связи.
- Дата последней редакции.
- Применимые законы: 152-ФЗ РФ (для резидентов РФ), законодательство страны пребывания субъекта ПДн (для нерезидентов).
- Ссылка на Оферту.

### 2. Какие ПДн мы собираем

Разделить на блоки по источнику получения.

**При регистрации:**
- email (обязательно — идентификатор);
- имя / имя отображения (опционально, по желанию);
- пароль (хранится в хэшированной форме, не доступен оператору в открытом виде).

**При использовании Сервиса:**
- IP-адрес (логируется для безопасности и защиты от злоупотреблений);
- user-agent браузера (технический параметр);
- загружаемый Пользователем контент (фото, комментарии, метаданные проектов);
- действия в Сервисе (переходы между этапами, комментарии, подтверждения) — для внутренней отчётности Заказчика.

**При оплате:**
- реквизиты платежа обрабатываются банком-эквайером, Maket CP получает только факт оплаты + сумму + идентификатор транзакции.

**Cookies и аналитика:**
- Session cookie для авторизации.
- Счётчик посещений (TBD — Yandex.Metrika или аналог) — указать, если используется.
- Нет трекеров для ретаргетинга в Q2. Если в Q3 появятся — обновляем Политику.

### 3. Цели обработки

- Предоставление Сервиса (основная цель).
- Аутентификация и защита от несанкционированного доступа.
- Обеспечение коммуникации с Заказчиком (email-уведомления, счёта, инциденты).
- Улучшение продукта на основе обезличенных агрегированных данных.
- Исполнение обязательств перед Заказчиком по Договору (Оферте).
- Исполнение требований законодательства (если регулятор запрашивает информацию в рамках установленной процедуры).

### 4. Правовые основания обработки (ст. 6, ст. 9 ФЗ-152)

Перечислить по пунктам, привязав к целям из §3.

- Согласие субъекта ПДн (при регистрации — явный checkbox «Я согласен с Политикой конфиденциальности»).
- Исполнение договора, по которому субъект ПДн — сторона (Оферта).
- Исполнение требований закона (хранение учётной документации).

### 5. Срок хранения

- ПДн Пользователя хранятся в течение срока действия Подписки + [X] дней grace period + [Y] дней архива (например, 6 месяцев для возможного восстановления учётной записи).
- По истечении архивного срока — удаление / обезличивание.
- Учётные документы (счета, акты) — хранятся в течение срока, требуемого НК РФ (5 лет).

### 6. Права субъекта ПДн

Ст. 14 ФЗ-152 + GDPR best practice.

- Право знать, какие данные обрабатываются.
- Право на доступ и получение копии.
- Право на исправление.
- Право на удаление («право быть забытым») — с оговорками по данным, которые оператор обязан хранить по закону.
- Право отозвать согласие (с предупреждением, что в этом случае Сервис прекращает быть доступным).
- Право на перенос данных (экспорт в машиночитаемом формате).
- Право на жалобу в Роскомнадзор.

**Форма реализации прав:** email на [{contact_email}]. Срок ответа — 30 дней.

### 7. Передача ПДн третьим лицам

- Никаких продаж ПДн третьим лицам.
- Передача субобработчикам — только в объёме, необходимом для оказания Сервиса:
  - **Supabase Inc. (США/ЕС, managed cloud)** — хранение данных, auth, storage. Регион размещения — указать актуальный (Frankfurt на 2026-04-23).
  - **OpenAI Inc. (США)** — если используется AI Vision. Передаются изображения без идентификатора субъекта (обезличенно), текст комментариев — нет.
  - **Платёжный провайдер [Тинькофф / ЮКасса / …]** — только реквизиты транзакции.
- Список актуализируется при изменении; уведомление — не менее чем за 30 дней через email / лендинг.
- Передача органам государственной власти — только по установленному законом запросу (постановление суда, предписание).

### 8. Международная передача данных

- Часть данных хранится на серверах Supabase Inc. в ЕС (Frankfurt).
- Условия передачи:
  - Согласие субъекта ПДн (активный checkbox при регистрации).
  - Договор с Supabase обеспечивает защиту данных в соответствии с GDPR (который сопоставим или строже 152-ФЗ по требованиям к безопасности).
- Для субъектов ПДн — резидентов РФ: данная передача осуществляется на основании вашего согласия.
- При появлении требований регулятора или по запросу Заказчика — возможна миграция в РФ-инфраструктуру (см. Оферту §8).

### 9. Безопасность

- HTTPS-шифрование всего трафика между Пользователем и Сервисом.
- Хранение паролей в хэшированной форме (bcrypt / argon2).
- Ограничение доступа к данным (Row-Level Security в Supabase, авторизация JWT).
- Резервные копии — регулярные, хранятся в ограниченном доступе.
- Мониторинг подозрительной активности.
- Уведомление о нарушении (в случае инцидента) — согласно 152-ФЗ + в разумный срок.

### 10. Обновления политики

- Исполнитель вправе обновлять Политику.
- Существенные изменения (новые категории ПДн, новые субобработчики, новое применимое право) — уведомление за 30 дней через email / баннер на лендинге.
- Несущественные изменения (корректировка формулировок, адресов, опечаток) — публикация обновлённой версии без предварительного уведомления.
- Все предыдущие версии сохраняются по публичной ссылке (можно сверить).

## Критичные места для review живым юристом

- 🚩 **§4 правовые основания** — формулировки должны строго соответствовать ст. 6 и ст. 9 152-ФЗ. Особенно — «согласие» (должно быть «конкретное, информированное, сознательное и добровольное»).
- 🚩 **§7 субобработчики** — перечень должен быть честным и актуальным. «Забытый» субобработчик = дыра в compliance.
- 🚩 **§8 международная передача** — ключевой риск-раздел. Юрист должен подтвердить формулировку, либо заменить на более консервативную.
- 🚩 **§5 срок хранения** — конкретные числа (30/180/365 дней) должны быть подкреплены функциональностью продукта. Если технически удаление через 180 дней — говорим 180.
- 🚩 **Обработчик vs оператор по Контенту Заказчика** — в Политике явно отразить, что в отношении ПДн, загружаемых Заказчиком (лица на фото), Исполнитель — обработчик по поручению Заказчика. Без этого получается, что мы оператор всех ПДн, включая лица на продуктовых фото, что технически не так.

## Открытые вопросы к Маше

1. **Email для privacy-запросов** — создать отдельный адрес {contact_email} или использовать barinski.m@gmail.com? Предлагаю первое для будущих клиентов.
2. **Cookies** — есть ли аналитика (Яндекс.Метрика / Google Analytics) в Q2? Если да — добавляем в §2.
3. **Срок архивного хранения** — после окончания подписки сколько дней держим ПДн перед удалением? Технически реализуем через триггер в Supabase (CoS / PA).
4. **OpenAI Vision** — включено ли в production? Если да — указываем как субобработчика. Если нет — не упоминаем.
5. **Уведомление об инцидентах** — какой SLA на коммуникацию к клиентам в случае утечки / инцидента? Стандарт GDPR — 72 часа, 152-ФЗ менее строг. Рекомендую 72 часа как best practice.

## Что НЕ включаем сейчас (парковка)

- **Полноценный cookie banner с гранулярным согласием** — стандарт GDPR, но для РФ-юрисдикции и B2B-сценария избыточно на старте. Активируется, когда появится EU-клиент или Яндекс.Метрика с расширенным трекингом.
- **Data Protection Officer (DPO)** — назначается при обязательствах, которые у нас сейчас не наступают. Парковка до enterprise-клиента, который запросит.
- **Отдельное уведомление о трансграничной передаче в Роскомнадзор** — по 152-ФЗ требуется. Осознанно пропускаем (см. `understanding-2026-04-23.md` §6 и принцип Маши). Активируется по триггеру.
