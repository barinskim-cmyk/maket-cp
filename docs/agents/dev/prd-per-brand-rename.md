> **TL;DR:** Каждый бренд приносит свои правила переименования. Сейчас имя файла зашито в коде (`{sku}_{num}.{ext}`). Нужен per-brand конфиг с шаблоном, переменными и UI для настройки. Этот PRD — только спецификация, без кода и миграций. Решение по data model: рекомендован вариант B (jsonb на проекте) как стартовый, с миграцией в A (отдельная таблица) на Phase 3.

# PRD — Per-brand rename logic

## 1. Background

Сегодня логика переименования зашита в `articles.js` (фронт) и `tools/maket_rename.py` (CSV-applier). Имя строится по жёсткой формуле `{sku}_{NN}.{ext}` — например, `PL01056CN-13-black-26S_01.jpg`. Это работает, пока продаваемое ядро обслуживает один сценарий (product cards с одним артикулом на карточку), но ломается при первом же бренде с другой конвенцией.

Реальные кейсы, которые уже маячат на горизонте: lookbook-съёмки (имя по луку, а не по SKU), кампании (дата + тема + номер), бренды с обязательным префиксом (короткий код бренда), бренды, которым нужен этап в имени (`_cc`, `_retouch`, `_final`), и бренды, у которых артикул содержит спецсимволы, недопустимые в их MAM (Media Asset Management). Адаптировать каждый раз правкой кода — дорого и хрупко: ломает других клиентов и не масштабируется.

Цель — вынести правило переименования из кода в конфиг бренда (или проекта), с возможностью настройки через UI и preview-проверкой.

## 2. User stories

1. **Owner бренда настраивает rename-правила.** Как owner [бренда X], я хочу зайти в settings бренда/проекта и задать шаблон имени файла, чтобы новые проекты использовали мою конвенцию без обращения к разработчику.
2. **При upload фото в проект бренда применяются его правила.** Как фотограф, ассистент или ретушёр, я хочу, чтобы при экспорте rename-CSV из артикул-инструмента имена файлов считались по правилам этого бренда автоматически — без ручного редактирования.
3. **Default-правило работает, если ничего не задано.** Как owner стартового бренда, я хочу не настраивать ничего и получать текущее поведение (`{sku}_{num}.{ext}`), чтобы простые случаи оставались простыми.
4. **Маша / админ может посмотреть и поменять правила вручную.** Как админ платформы, я хочу видеть конфиг переименования любого бренда и при необходимости править его (например, после жалобы клиента), чтобы быстро гасить инциденты без релиза.
5. **Owner проверяет правило на preview перед сохранением.** Как owner, я хочу видеть, как будут названы 3 примера фото при текущем шаблоне, чтобы поймать опечатку до того, как переименую сотни файлов.

## 3. Data model

### Option A — отдельная таблица `brand_rename_rules`

```
brand_rename_rules:
  - id uuid pk
  - brand_id uuid fk → brands (или owner_id если таблицы brands нет)
  - rule_name text          (e.g. "default", "lookbook", "campaign-2026")
  - pattern_template text   (e.g. "{brand_short}_{article}_{stage}_{num}")
  - separators jsonb        (опц.: разделители, padding, case)
  - is_default boolean
  - created_at, updated_at timestamptz
  - constraint: один is_default=true на brand_id
```

**Pros:** правил несколько на бренд (default + lookbook + кампания), нормализованная структура, легко искать «у каких брендов нестандартный rename», аудит изменений по строке.
**Cons:** требует таблицы `brands` (которой пока нет — есть `profiles` + неявная связь через `projects.owner_id`); миграция тяжелее; UI должен уметь выбирать «какое правило применить» при экспорте.

### Option B — jsonb-поле на `projects`

```
projects.rename_config jsonb default '{}'::jsonb
-- пример содержимого:
{
  "version": 1,
  "template": "{brand_short}_{article}_{num}",
  "padding": 2,
  "case": "upper",
  "separator": "_",
  "vars": { "brand_short": "MF" }
}
```

**Pros:** ноль миграций кроме одного `ALTER TABLE`; конфиг живёт там же, где проект (легко клонировать «по образцу прошлой съёмки»); если правило одно на проект — этого хватает; не требует таблицы `brands`.
**Cons:** дублирование при 10 проектах одного бренда (правят в одном — забывают в других); нет наследования «бренд → проект»; сложно отвечать на вопрос «покажи все бренды с правилом X».

### Option C — поле на `profiles` (owner = бренд)

```
profiles.rename_template text
profiles.rename_vars jsonb
```

**Pros:** одна точка правды на owner-а, минимум схемы.
**Cons:** ломается, как только owner ведёт два бренда с разной конвенцией (Маша уже фотографирует под несколько брендов — это не гипотетика); не выживет в multi-brand portfolio-агентствах.

### Рекомендация

**Option B** на старт + миграция к **Option A** на Phase 3, когда появится сущность `brands` (она уже намечена в `project_brand_client_continuity`).

Обоснование:
- Стартовый сценарий — *один проект = один бренд*. JSON-поле покрывает 100 % текущих кейсов без новой таблицы.
- ALTER TABLE с `default '{}'::jsonb` безопасен для существующих проектов: пустой объект → работает default-правило (текущий хардкод).
- Когда придёт первый owner с двумя брендами или появится таблица `brands`, переезд на A — это `INSERT INTO brand_rename_rules SELECT ... FROM projects` + UI-переключатель «правило бренда / правило проекта». Дешевле, чем построить A сразу и использовать на 5 %.
- Принцип «рост > формальность» (см. CLAUDE.md, project memory).

## 4. Pattern syntax

Шаблон — строка с переменными в фигурных скобках. Парсер делает простую подстановку, без вычислений.

### Базовые переменные

| Переменная             | Источник                                | Пример                  |
|------------------------|-----------------------------------------|-------------------------|
| `{brand_short}`        | `rename_config.vars.brand_short`        | `MF`                    |
| `{brand_full}`         | `rename_config.vars.brand_full` или `projects.brand` | `Maket Fashion` |
| `{article}` / `{sku}`  | `articles.sku`                          | `A12345`                |
| `{stage}`              | этап из pipeline (`cc`, `retouch`, `final`) | `retouch`           |
| `{num}`                | номер позиции в карточке (с padding)    | `001`                   |
| `{date}`               | `projects.shoot_date` (формат `YYYY-MM-DD`) | `2026-04-25`        |
| `{photographer_short}` | инициалы фотографа                      | `MB`                    |
| `{lookbook_id}`        | id лука (для lookbook-сценариев)        | `LB07`                  |
| `{ext}`                | расширение исходного файла              | `jpg`                   |

### Модификаторы

- `{var:upper}` / `{var:lower}` — регистр (override глобальной настройки `case`).
- `{num:3}` — padding (override глобальной `padding`).
- `{date:YYYY-MM}` — формат даты (минимальный набор: `YYYY`, `MM`, `DD`, разделители `-` `_`).

### Глобальные настройки

```
padding:   number   (default 2 — например, 01, 02 ... 99)
case:      "asis" | "upper" | "lower" (default "asis")
separator: string   (default "_") — для технических вставок, не заменяет {} в шаблоне
ext_case:  "asis" | "lower" (default "lower" — .JPG → .jpg)
```

### Примеры

| Сценарий                 | Шаблон                                          | Результат                          |
|--------------------------|-------------------------------------------------|------------------------------------|
| Default (текущий)        | `{sku}_{num}`                                   | `PL01056CN-13-black-26S_01.jpg`    |
| С брендом                | `{brand_short}_{article}_{stage}_{num}`         | `MF_A12345_retouch_001.jpg`        |
| Lookbook                 | `{date}_{lookbook_id}_{num}`                    | `2026-04-25_LB07_003.jpg`          |
| Кампания                 | `{brand_short}_camp{date:YYYY-MM}_{num:3}`      | `MF_camp2026-04_007.jpg`           |
| С фотографом             | `{brand_short}_{photographer_short}_{date}_{num}` | `MF_MB_2026-04-25_01.jpg`        |

### Валидация шаблона

- Шаблон должен содержать хотя бы одну переменную, гарантирующую уникальность (`{num}`, `{article}`, `{lookbook_id}`).
- Запрещённые символы для имени файла (по сводному списку Win/macOS/Linux): `\ / : * ? " < > |` — отрезаются в результате с подстановкой `_` (но шаблон с такими символами в литералах система блокирует на сохранении).
- Кириллица в результате допустима технически, но *не рекомендуется* — Storage уже горел на этом (см. `incident_ekonika_cyrillic_filenames`). UI показывает предупреждение, если в `vars` или литералах шаблона есть не-ASCII.
- Длина итогового имени < 200 символов (Storage limit запас).

## 5. UI flow

Расположение: **Settings проекта → секция «Имена файлов»** (Phase 2). На Phase 3, когда появится сущность `brands`, дублируется в Settings бренда с переключателем «использовать правило бренда / переопределить для проекта».

```
[ Имена файлов ]

Шаблон:  [ {brand_short}_{article}_{stage}_{num}            ]   [Сбросить к default]

Переменные бренда:
  brand_short:  [ MF       ]
  brand_full:   [ Maket Fashion ]
  photographer_short: [ MB ]   (опц.)

Настройки:
  Padding:    [ 2 ▾ ]
  Регистр:    ( ) Как есть   (•) UPPER   ( ) lower
  Расширения: (•) lowercase   ( ) Как есть

Превью на 3 примерах:
  DSC_0042.jpg    →  MF_A12345_RETOUCH_01.jpg
  DSC_0043.jpg    →  MF_A12345_RETOUCH_02.jpg
  DSC_0044.jpg    →  MF_A12346_RETOUCH_01.jpg

[ Сохранить ]   [ Отмена ]
```

Поведение:
1. При открытии — текущий шаблон бренда/проекта или default (`{sku}_{num}`), если ничего не задано.
2. Любое изменение шаблона/настроек / переменных → preview пересчитывается на лету (берёт 3 «случайных» назначенных пары `(card, slot)` из последнего экспорта артикулов; если экспортов не было — синтетический пример).
3. Запрещённые символы или невалидный шаблон → сохранить нельзя, под полем красное сообщение с конкретной причиной.
4. Save → новые экспорты CSV из артикул-инструмента используют новый шаблон. Уже сгенерированные CSV-файлы и уже переименованные файлы НЕ трогаем (см. п.6 «Migration path» и п.9 «Risks»).
5. Кнопка «Сбросить к default» — мягкая, требует подтверждения, сохраняет старый шаблон в `rename_config.previous_template` для отката.

Доступ (роли — финальный список — open question 4):
- Owner проекта: видит и редактирует.
- Админ платформы (Маша): видит и редактирует у любого проекта.
- Клиент / ретушёр: видит, не редактирует.

## 6. Migration path

**Существующие проекты.** При выкатке Phase 1: `ALTER TABLE projects ADD COLUMN rename_config jsonb NOT NULL DEFAULT '{}'::jsonb`. Backend проверяет: если `rename_config` пустой объект — применяет default-правило (`{sku}_{num}`), идентичное текущему хардкоду. Никаких визуальных или поведенческих изменений для существующих owner-ов.

**Новые проекты.** В диалоге создания проекта (Phase 2) появляется опциональный шаг «Имена файлов»: выбор «использовать default» или «настроить». По умолчанию выбран default, чтобы не утяжелять onboarding.

**Миграция уже переименованных файлов.** НЕ переименовываем задним числом. Изменение шаблона действует только на следующие экспорты CSV из артикул-инструмента. Это решение фиксируется в UI: предупреждение «изменение применится только к новым экспортам, ранее переименованные файлы остаются как есть».

**Backwards compat для скрипта `maket_rename.py`.** Скрипт уже работает с готовым CSV (`old_name,new_name`) и не знает о шаблонах — менять его не нужно. Вся логика шаблона — на стороне веб-клиента / backend при экспорте.

**Откат.** Если шаблон сломал генерацию (например, дал коллизии) — UI кнопка «Сбросить к default» восстанавливает поведение. История шаблонов хранится либо в `rename_config.history[]`, либо (Phase 3) отдельной таблицей `brand_rename_rules` с soft-delete.

## 7. Implementation phases

### Phase 1 — Q3 (data model + parser, без UI)

- ALTER `projects ADD COLUMN rename_config jsonb DEFAULT '{}'::jsonb`.
- Backend / frontend: helper `buildFileName(template, vars, globalSettings)` с поддержкой базовых переменных и модификаторов из п.4.
- Подключить helper в `articles.js → downloadCSV()` вместо хардкода.
- Тесты: 10–15 unit-тестов на парсер (default, all vars, edge cases — пустые vars, спецсимволы, длина).
- Hardcoded preview в DevTools (без UI), чтобы разработчик мог проверить шаблон вручную через `localStorage`.
- **Effort: ~2–3 дня.** (Парсер + helper + миграция + smoke-тест на одном проекте.)

### Phase 2 — Q3 (UI в Settings проекта)

- Секция «Имена файлов» в settings проекта по макету из п.5.
- Live-preview на 3 примерах.
- Валидация шаблона на клиенте + повторно на backend при сохранении.
- Опциональный шаг «Имена файлов» в визарде создания нового проекта.
- Документация для owner-ов (короткая страница в landing/help).
- **Effort: ~4–6 дней.** (UI + валидация + preview + копирайт + smoke-тест с участием Маши.)

### Phase 3 — Q4 (multi-rule + brands)

- Таблица `brand_rename_rules` (по схеме Option A из п.3).
- Сущность `brands` или соглашение о её роли (зависит от прогресса по `project_brand_client_continuity`).
- Поддержка нескольких правил на бренд (default + lookbook + кампания) с явным выбором при экспорте.
- Поддержка multi-article карточек (lookbook: одна карточка → несколько артикулов → один файл) — формат `{articles_joined}` или явный шаблон с циклом.
- Миграция данных из `projects.rename_config` в `brand_rename_rules` (one-time job).
- **Effort: ~7–10 дней.** (Зависит от готовности `brands`; если её ещё нет — больше.)

## 8. Open questions

1. **Variable namespace strict или flexible?** Закрытый список переменных (только то, что в п.4) или owner может объявлять свои `vars` любого имени и подставлять? Strict — безопаснее и проще валидировать; flexible — гибче для нестандартных кейсов (например, `{collection_code}`, `{season}`). Рекомендация PA: strict на Phase 1–2, flexible через `vars`-словарь на Phase 3.
2. **Validation: regex или free-form?** Помимо проверки спецсимволов, нужна ли возможность задать regex для валидации артикула (например, бренд требует, чтобы SKU был всегда `[A-Z]{2}\d{5}`, и UI должен ругаться при загрузке несоответствующего)? Это уже не rename, а валидация артикулов — может быть отдельным PRD.
3. **Per-stage правила или единое?** Нужны ли разные шаблоны на разных этапах (cc / retouch / final), или достаточно одного шаблона с переменной `{stage}`? PA склоняется ко второму (проще и покрывает 90 %), но клиенты с раздельными FTP-папками под этапы могут хотеть разное.
4. **Кто может менять правила?** Owner only? Owner + админ платформы? Owner + ретушёр (в особых случаях)? И главное — нужен ли audit trail (кто и когда поменял шаблон), чтобы расследовать «почему вдруг пошли файлы с другим именем»?
5. **Конфликты при ре-экспорте.** Если owner изменил шаблон, потом снова экспортирует CSV для того же проекта — что мы делаем? Перегенерируем имена с нуля (риск дублей при ручном переименовании) или дописываем только новые? Рекомендация PA: перегенерируем всё, но в CSV добавляем колонку `existing_new_name`, чтобы скрипт пропускал уже переименованные файлы.

## 9. Risks

- **Несовместимые имена для одних и тех же фото.** Owner поменял шаблон mid-project → часть файлов переименована по старому, часть по новому. Митигация: предупреждение в UI, журнал переименований (`rename_log` уже есть — см. `supabase.js:4616`), чтобы было видно, какой шаблон применялся к каждому файлу.
- **Коллизии имён.** Шаблон без `{num}` или с малым padding → два разных файла получают одно имя → второй CSV-row перезапишет первый при apply. Митигация: парсер обязан проверять уникальность результата на этапе генерации CSV и блокировать экспорт с человекочитаемой ошибкой («3 файла получили имя `MF_A12345.jpg`, добавьте `{num}` в шаблон»).
- **Performance при больших batch'ах.** Бренд с 5000 фото и сложным шаблоном с regex-валидацией → генерация CSV в браузере подвиснет. Митигация: парсер — простая string substitution без regex; benchmark на 10 000 файлов в Phase 1.
- **UX-сложность для non-tech owners.** Шаблон с фигурными скобками и модификаторами — это «программирование» для непрограммиста. Митигация: пресеты («Default», «С брендом», «С датой», «Lookbook»), preview на лету, всплывающая подсказка с примерами при наведении на `{var}`.
- **Storage limits (Cyrillic, длина).** Storage уже горел на кириллице (incident `ekonika_cyrillic_filenames`). Митигация: жёсткое предупреждение и `sbSanitizeStorageKey`-стиль санитизация на финальном результате, а не только на шаблоне.
- **Изменения шаблона ломают share-link.** Если файл уже расшарен по старому имени, а его переименовали — клиент видит 404. Митигация: share-link должен резолвить по UUID, не по имени файла (это уже так в текущей архитектуре, но нужно подтвердить QA).
- **Стоимость поддержки многообразных шаблонов в saaS-режиме.** Каждый «креативный» owner = поддержка одного edge-case. Митигация: 4 пресета покрывают 80 %, custom — на свой риск с отказом от guarantee при экзотических шаблонах.
