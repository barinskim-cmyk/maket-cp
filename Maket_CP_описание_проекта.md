## 1. Цель проекта

Maket CP — единый инструмент для управления полным циклом коммерческой фотосъёмки: от первого отбора до сдачи готового материала клиенту. Проект решает ключевую проблему: сейчас фотографы используют множество разрозненных инструментов (Capture One, мессенджеры, Excel, Figma) и теряют часы на ручной перенос данных между ними.

### 1.1. Ключевая идея

Карточка товара — центральная сущность системы. Это не просто раскладка фото, а творческий инструмент — позволяет увидеть, как кадры работают вместе. Карточка рождается на этапе отбора и проходит через весь пайплайн: согласование, коррекция, ретушь, адаптация, сдача.

### 1.2. Целевые пользователи

- **Фотографы** (основной) — коммерческие фотографы, работающие с брендами одежды, аксессуаров, обуви
- **Клиенты** (бренды) — команды от 1 до 4+ человек, согласовывающие отбор
- **Ретушёры** — получают ТЗ с комментариями, отдают результат

### 1.3. Целевой масштаб

- Типичная съёмка: 2000+ фотографий, 300+ после отбора
- Типичный проект: 15–50 товаров (карточек)
- Каналы публикации: маркетплейсы (WB, Ozon, Lamoda), собственный сайт бренда, соцсети

### 2.1. Текущие болевые точки

## 4. Техническое решение (v2.1)

### 4.1. Архитектурный принцип

Desktop-first web app с Python engine. UI полностью на HTML/JS — единая кодовая база для десктопа и веба. Python выступает как application engine + API слой (через pywebview bridge). Система работает с сущностями (Photo, Card, Project, Article), а не с файлами напрямую.

Десктоп и веб — единый flow: авторизация через Supabase, непрерывная синхронизация, локальное хранение при отсутствии интернета с автоматической синхронизацией при восстановлении соединения.

```
Frontend (index.html + модули js/*.js)
        ↓
window.pywebview.api.*
        ↓
AppAPI (Python, имеет доступ к window)
        ↓
Service Layer (бизнес-логика)
        ↓
Infrastructure (файлы, Capture One .cos)
```

### 4.2. Архитектурные решения (v2.1)

Ключевые решения, принятые при переходе от v1 (скрипты) к v2.1 (система):

| Решение | Было (v1) | Стало (v2.1) | Почему |
|---------|-----------|-------------|--------|
| UI | tkinter | HTML/JS через pywebview | Единая кодовая база desktop + web |
| File dialogs | `tkinter.filedialog` | `window.create_file_dialog()` | Консистентность — всё через pywebview |
| Progress updates | — | `window.evaluate_js()` push | Event-driven, realtime, без polling |
| Frontend модули | card-maker.html = self-contained | Модульная архитектура: index.html + js/state.js, nav.js, shootings.js, cards.js, sync.js | Разделение ответственности, удобство поддержки |
| API регистрация | — | `js_api=api` в `create_window` | Стабильнее в PyInstaller, чем `window.expose()` |
| Мышление | Файлы | Сущности (Photo, Card, Project) | Масштабируемость, бизнес-логика отдельно от I/O |

### 4.3. Слоистая архитектура

#### Domain Layer — Сущности системы

Ключевой сдвиг: от «скриптов, работающих с файлами» к «системе, работающей с сущностями».

```python
class Photo:
    name: str       # исходное имя файла
    stem: str       # имя без расширения/суффиксов
    path: Path      # путь к файлу


class Slot:
    """Позиция в карточке. Содержит варианты (опции) фото."""
    position: int              # порядковый номер в карточке
    aspect: str                # "3:2" | "2:3"
    options: list[Photo]       # варианты фото для этой позиции
    selected: Photo | None     # выбор клиента (галочка)
    comment: str | None        # комментарий клиента ("хочу ещё вариант")


class Card:
    id: str
    category: str
    slots: list[Slot]          # позиции с вариантами (вместо плоского list[Photo])
    status: str                # draft, pending, approved, done
    comments: list[Comment]    # общие комментарии к карточке


class SlotDef:
    """Определение слота в шаблоне."""
    aspect: str                # "3:2" | "2:3"
    label: str | None          # "главное фото", "деталь" и т.п.


class CardTemplate:
    """Шаблон карточки — задаётся при создании проекта."""
    name: str                  # "горизонт + 3 вертикали"
    slots: list[SlotDef]       # [{aspect: "3:2"}, {aspect: "2:3"}, ...]


class Project:
    brand: str
    shoot_date: str
    template: CardTemplate     # шаблон карточки для проекта
    cards: list[Card]
```

#### API Layer — Bridge между JS и Python

AppAPI получает ссылку на `window` при инициализации. Это даёт доступ к нативным диалогам (`create_file_dialog`) и push-событиям (`evaluate_js`).

```python
class AppAPI:
    """Регистрируется в pywebview, доступен из JS как window.pywebview.api.*"""

    def __init__(self, window):
        self.window = window
        self.rate_setter = RateSetterService()

    # ── Системные ──
    def ping(self) -> str:
        return "pong"

    def select_folder(self) -> str | None:
        result = self.window.create_file_dialog(webview.FOLDER_DIALOG)
        return result[0] if result else None

    # ── Rate Setter ──
    def rate_setter_run(self, payload: dict) -> dict:
        """Запуск в отдельном потоке с push-прогрессом."""
        import threading

        def task():
            stems = payload.get("stems", [])
            session_path = Path(payload.get("session_path"))
            result = self.rate_setter.run(stems=stems, session_root=session_path)
            self.window.evaluate_js(f"window.onRateSetterDone({json.dumps(result)})")

        threading.Thread(target=task).start()
        return {"status": "started"}

    # ── Проект ──
    def load_project(self) -> dict: ...
    def save_project(self, data: dict) -> dict: ...
```

### 4.4. Взаимодействие Frontend ↔ Backend

#### Общий поток вызова

```
[Frontend UI] → window.pywebview.api.method(payload)
        ↓
[AppAPI]      → валидация, маршрутизация
        ↓
[Service]     → бизнес-логика
        ↓
[Infra]       → файловая система, .cos
        ↓
[AppAPI]      → window.evaluate_js() — push результата в UI
        ↓
[Frontend UI] → обновление интерфейса
```

#### Реальный пример: Rate Setter

**JS (frontend):**
```javascript
async function runRateSetter() {
    const payload = {
        stems: ["IMG_0001", "IMG_0002"],
        session_path: "/Users/masha/session"
    };
    await window.pywebview.api.rate_setter_run(payload);
}

// Push-события из Python:
window.onProgress = (progress) => {
    document.getElementById('progress').style.width = progress + '%';
};
window.onRateSetterDone = (result) => {
    alert(`Готово! Обновлено: ${result.updated}, не найдено: ${result.missing}`);
};
```

**Python (backend):**
```python
def rate_setter_run(self, payload):
    def task():
        stems = payload["stems"]
        total = len(stems)
        for i, stem in enumerate(stems):
            # обработка...
            progress = int((i / total) * 100)
            self.window.evaluate_js(f"window.onProgress({progress})")

        result = {"updated": 10, "missing": 2}
        self.window.evaluate_js(f"window.onRateSetterDone({json.dumps(result)})")

    threading.Thread(target=task).start()
    return {"status": "started"}
```

#### Event-driven подход (вместо polling)

Вместо `setInterval` для проверки статуса — push из Python через `window.evaluate_js()`. Это даёт realtime UI без лишних запросов и правильный event-driven flow.

```python
# Python: push-события
def _emit(self, event_name: str, data):
    """Универсальный push из Python в JS."""
    self.window.evaluate_js(f"window.{event_name}({json.dumps(data)})")
```

### 4.5. Компоненты системы

#### Card Maker — Вёрстка и согласование карточек товара ✅ / 🔧

Визуальный редактор карточек товара. Ядро системы. Не просто инструмент вёрстки — полноценный инструмент согласования отбора.

**Текущее (v1):**
- Шаблон: 1 горизонтальное фото (3:2) + 2–3 вертикальных (2:3)
- Drag-and-drop фото из папки на компьютере прямо в браузер
- Перемещение фото между слотами (swap)
- Категории: очки, сумки, аксессуары, обувь
- Экспорт: PDF, список фото (.txt), сохранение проекта как HTML
- Автосохранение через IndexedDB, возобновление сессии
- Модульная архитектура: cards.js (логика) + style.css (стили) + index.html (разметка)

**Новое (v2.1) — Настраиваемые шаблоны:**
- Фотограф задаёт шаблон карточки при создании проекта
- Предустановки: «4–7 вертикалей», «горизонт + 2–3 вертикали», «горизонт + 4–5 вертикалей»
- Возможность создать кастомный шаблон (количество слотов, пропорции каждого)
- Шаблон применяется ко всем карточкам проекта

**Новое (v2.1) — Опции (варианты) в слоте:**
- Слот — это не «одно фото», а «позиция с вариантами»
- Фотограф перетаскивает несколько фото в один слот — они становятся опциями
- В UI: карусель/свайп внутри слота для просмотра вариантов
- Клиент может: выбрать вариант (галочка), удалить лишний, запросить дополнительный вариант с комментарием
- Индикатор на слоте: «3 варианта» — чтобы было видно, где есть опции

#### Rate Setter — Сервис (без собственного UI) ✅

Автоматическое выставление рейтингов в Capture One. Вызывается из web UI через bridge.

- Читает список утверждённых фото (из папки или текстового списка)
- Находит .cos файлы в сессии Capture One, выставляет 5★
- Умное удаление расширений и суффиксов (_preview, _web, _copy)
- Экономит часы ручного поиска фотографий в каталоге
- Атомарная запись + бэкап .cos.bak + dry-run режим

#### Maket CP App — Application Shell 🔧

- Загрузка UI (HTML), регистрация Python API через `js_api`
- Навигация между модулями через `load_url()` / `load_html()`
- JS–Python мост через pywebview (push-события через `evaluate_js`)
- Логирование, state management
- Модульная архитектура: легко добавлять новые вкладки

### 4.9. Безопасность и надёжность

- **Атомарная запись:** tmp.write() → tmp.replace() — предотвращает битые файлы при сбое
- **Бэкапы:** .cos.bak перед каждым изменением
- **Dry-run режим:** показать что будет изменено, без записи
- **Кеш cos_index:** ускорение повторных запусков, проверка через mtime

```
   ├── share_links                 — токены доступа по ролям (client/retoucher)
   ├── comments, stage_events      — история и комментарии
   └── Storage buckets             — thumb + preview изображения
```

Синхронизация: непрерывная двусторонняя. При каждом изменении карточки/слота — авто-синхронизация с Supabase (debounce 2-3 сек). При отсутствии интернета — сохранение в localStorage + IndexedDB, автоматическая синхронизация при восстановлении (событие `online`). Новые проекты автоматически загружаются в облако при первом сохранении.

Share-ссылки: анонимный доступ по токену (?share=TOKEN). RLS-политики Supabase разделяют доступ owner/member/anonymous. Клиент может редактировать карточки через RPC-функцию (без авторизации).

- Переход по реальным триггерам (не кнопка "Завершить"), фиксация времени перехода

- Изменяемый размер миниатюр (ползунок)

- Список обновляется при сохранении CP/доп. отбора, Rate Setter читает готовый список

**Доп. контент:**
- Теги площадок публикации (WB, Ozon, соцсети) для этапа адаптации
- Разделение: отбор фотографа vs. доп. задание (баннеры, соцсети)
