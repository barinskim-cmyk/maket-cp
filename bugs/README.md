# Maket CP — Bug Tracker (файловый)

**Статус:** активный процесс.
**Владелец процесса:** QA (реализация), CoS (SLA, эскалация), CX (первичный triage).
**Инициализирован:** 2026-04-23 overnight, QA lane.

Эта папка — единственный источник правды для дефектов Maket CP до момента, когда будет подключён полноценный bug tracker (Linear/YouTrack). Пока команда небольшая и большая часть сигнала приходит из live-тестов, живых клиентов и chrome-agent'ов, сверхлёгкий файловый процесс эффективнее, чем trackers с ролями и правами.

Любой агент (CX, QA, PA, chrome-agent, CoS) или Маша создаёт файл в этой папке при обнаружении дефекта. Ничего «позже сведу в Linear» — единый формат с самого начала.

---

## 1. Формат имени файла

```
YYYY-MM-DD-short-slug.md
```

- Дата — день обнаружения, а не день репорта (чтобы SLA отсчитывался от реальности).
- Slug — 3-6 слов через дефисы, английские буквы и цифры.
- Допустимо: `2026-05-03-rate-setter-sync-{domain_hidden}`
- Допустимо: `2026-05-03-share-link-404-on-revoke.md`
- Недопустимо: `bug1.md`, `urgent_fix.md`, `RateSetterBug.md`.

Если в один день несколько багов в одной области — slug всё равно должен быть различимым (добавь техническую деталь: `rate-setter-sync-cyrillic`, `rate-setter-sync-duplicate-upload`).

## 2. Структура bug-файла

Каждый файл обязан начинаться с YAML front-matter и содержать все девять секций ниже. Пустые секции заполняются как `— (нет данных)` — это лучше, чем их отсутствие, потому что даёт QA понять, что поле было осознанно пропущено, а не забыто.

### 2.1 YAML front-matter

```yaml
---
id: BUG-2026-001
date: 2026-05-03
reporter: [test user A]
severity: critical
area: rate-setter-sync
status: open
source: live-test
---
```

**Поля:**

- **`id`** — формат `BUG-YYYY-NNN`, последовательная нумерация внутри года. Следующий свободный номер определяется по максимальному `id` в этой папке за текущий год + 1. Если два агента параллельно создали один и тот же номер — приоритет у более ранней даты/времени коммита, второй перенумеровывает.
- **`date`** — день обнаружения, ISO-формат.
- **`reporter`** — имя человека или агентская роль: `Masha`, `[test user A]`, `CX`, `QA`, `PA`, `chrome-agent`, `CoS`, `client-<имя-клиента>`.
- **`severity`** — `critical` | `high` | `medium` | `low`. Определения ниже, см. раздел 3.
- **`area`** — функциональная область из закрытого списка (раздел 4). Если твоя область не из списка — сначала добавь её в список через PR, потом используй.
- **`status`** — `open` | `in-triage` | `in-progress` | `fixed` | `verified` | `closed` | `reopened` | `wontfix`. Определения — раздел 5.
- **`source`** — откуда пришёл сигнал: `live-test` | `chrome-agent` | `playwright` | `manual` | `client-reported` | `internal-review`.

### 2.2 Секции markdown

Ниже front-matter, в строго следующем порядке (чтобы гreppабельность сохранялась):

```markdown
## Summary
Один абзац (2-4 предложения). Что сломано, у кого, при каких условиях. Без скриншотов, без логов — этого достаточно CoS'у для оценки серьёзности за 30 секунд.

## Steps to reproduce
1. Пошаговая инструкция.
2. Каждый шаг = одно действие.
3. С указанием конкретного data (project_id, имя файла, URL), а не «какой-нибудь проект».

## Expected
Что должно было произойти. Ссылка на спеку/дизайн/test-план если есть.

## Actual
Что произошло. Точная формулировка ошибки, текст alert'а, exit code.

## Impact
- Сколько пользователей затронуто (оценка).
- Сколько проектов / файлов / данных затронуто.
- Есть ли потеря данных (если да — это всегда critical).
- Есть ли workaround (если да — описать).
- Блокирует ли бета-онбординг (если да — CoS запускает T-01, см. раздел 7).

## Logs / attachments
- Ссылка на лог в Supabase logs (таймкод UTC).
- Путь к скриншоту / gif / видео (в `docs/agents/qa/attachments/<bug-id>/`).
- Выжимка stack trace — не весь log, только релевантные 10-30 строк.

## Timeline
Список с датами и ISO-временем (UTC). Минимум:
- `YYYY-MM-DD HH:MM` — reported by <reporter>
- `YYYY-MM-DD HH:MM` — triaged by CX, severity=<x>, area=<y>
- `YYYY-MM-DD HH:MM` — reproduced by QA
- `YYYY-MM-DD HH:MM` — in-progress by PA (<имя>)
- `YYYY-MM-DD HH:MM` — fixed, commit <hash>
- `YYYY-MM-DD HH:MM` — verified by QA
- `YYYY-MM-DD HH:MM` — closed

Обновлять timeline при каждом изменении `status`. Это единственный аудируемый след процесса — не ломать.

## Resolution
Заполняется, когда `status` переходит в `fixed` или `wontfix`. Один-два абзаца:
- Что было причиной (root cause).
- Как починили (или почему wontfix).
- Какая регрессия покрыта тестом (ссылка на `tests/e2e/...` или пометка «TODO: test pending»).
- Что ещё потенциально затронуто (scope of collateral).

## Related
Ссылки на другие bug-файлы, PR'ы, спеки, incident-отчёты. Опционально.
```

## 3. Severity — определения

Severity ставится **исключительно** по impact на пользователя, не по сложности фикса и не по настроению reporter'а. Если сомневаешься — ставь на уровень выше: апгрейд проще, чем эскалация позже.

### 3.1 `critical`
- Блокирует использование продукта для ≥1 живого клиента или ≥1 активного проекта.
- Теряются данные (карточки, фото, рейтинги, комментарии).
- Security / privacy breach (утечка данных между тенантами, RLS обход, публичный доступ к приватному проекту).
- Data corruption в БД (противоречивые состояния, которые нельзя откатить из backup'а без ручного вмешательства).

**Автоматическое действие:** CoS запускает trigger T-01 — бета-поток (новые beta-заявки, outbound) ставится на паузу до перехода бага в `fixed`. См. раздел 7.

### 3.2 `high`
- Основной feature сломан, но есть workaround (пусть и неудобный).
- Пример: auto-rename падает на кириллических именах, но работает на латинице.
- Сломан один из критических смотровых флоу (owner dashboard, share-link открытие гостем, rate-setter sync).

### 3.3 `medium`
- Неудобство, заметное пользователю, но не блокирующее основную работу.
- Визуальный bug, который мешает читать UI, но не ломает логику.
- Пример: preview panel не ресайзится мышью, но ширину можно задать через скрипт.

### 3.4 `low`
- Косметика (отступы, цвета, опечатки).
- Полезно починить, но не страдает никто, кроме эстетики.

## 5. Status — lifecycle

```
open  →  in-triage  →  in-progress  →  fixed  →  verified  →  closed
                                         ↓
                                     reopened  (→ обратно в in-progress)
                                         
open  →  wontfix  (терминальный, с обоснованием в Resolution)
```

Определения:

- **`open`** — репортер только что создал файл. Никто ещё не смотрел.
- **`in-triage`** — CX поставил severity и area, подтвердил, что это правда баг, а не misuse. Передал QA.
- **`in-progress`** — PA взял в работу, идёт fix.
- **`fixed`** — PA запушил коммит, SOL указан в Resolution.
- **`verified`** — QA проверил на staging / prod после fix'а, баг реально не воспроизводится.
- **`closed`** — тест покрытия добавлен (или осознанно пропущен с пометкой), bug закрыт навсегда.
- **`reopened`** — QA или клиент воспроизвёл тот же баг после `fixed`/`closed`. Новый timeline entry, возвращаемся в `in-progress`.
- **`wontfix`** — команда решила не чинить (обязательно — обоснование в Resolution: out-of-scope, слишком дорого, duplicate, not a bug).

## 6. SLA (по severity)

Считаются от момента, когда `status` стал `open` (первая строка timeline). Нарушенные SLA — предмет еженедельного обзора CoS.

| Severity | Triage start | Fix start | Resolution target |
| -------- | ------------ | --------- | ----------------- |
| critical | < 2 часа     | < 4 часа  | < 24 часа         |
| high     | < 8 часов    | < 24 часа | < 72 часа         |
| medium   | < 24 часа    | — (по приоритетам команды) | < 1 неделя |
| low      | < 1 неделя   | backlog   | backlog           |

Если SLA под угрозой (например, критикал открыт 1.5 часа без triage'а) — CoS эскалирует Маше в Telegram, не в файл.

---

## 7. Trigger T-01 — пауза бета-потока

Условие: любой bug с `severity: critical` в `status: open` | `in-triage` | `in-progress`.

Автоматическое действие CoS (до полной автоматизации — ручное, но детерминированное):

1. Остановить отправку outbound писем «приглашение в бету».
2. Поставить форму бета-заявки в режим «мы ставим бету на короткую паузу» (короткий баннер, не disable).
3. Уведомить команду в основном канале (или Маше в Telegram): «T-01 активирован по BUG-YYYY-XXX».
4. Снять паузу **только** когда баг перешёл в `fixed` и есть план verify в течение 24 часов.

T-01 не применяется при `high`/`medium` — нельзя останавливать онбординг из-за косметики.

## 8. Ролевая модель

| Роль          | Зона ответственности в процессе багов |
| ------------- | ------------------------------------- |
| **Reporter**  | Создаёт файл, заполняет все обязательные секции. Не ставит severity — оставляет поле пустым или ставит свою догадку. |
| **CX**        | Первичный triage: читает Summary, подтверждает воспроизводимость по Steps, ставит финальный severity и area, переводит в `in-triage`. |
| **QA**        | Подтверждает воспроизводимость на staging. После fix'а — проверяет, добавляет e2e-покрытие в `tests/e2e/pending/` (или активирует существующий draft, если готов). |
| **PA**        | Реализует fix. Обновляет timeline и Resolution. Коммитит в main. |
| **CoS**       | Отслеживает SLA, запускает T-01, эскалирует Маше. Еженедельный review нарушений. |
| **Masha**     | Может репортить баг, может менять приоритеты, может закрывать `wontfix` с обоснованием. Не обязана делать triage сама. |

## 9. Что НЕ попадает в `/bugs/`

- **Фича-реквесты** — в `pipeline_backlog.md` или `tasks/`.
- **Вопросы** — в `docs/agents/<роль>/questions/`.
- **Гипотезы «может быть баг, но не воспроизвёл»** — в `docs/agents/qa/hypotheses/`.
- **Инциденты** (massive data loss, security breach с прод-последствиями) — в `incident_<...>.md` в корне + cross-ref в `/bugs/`.

Если не уверен, где должна жить запись — создай в `/bugs/` со `status: in-triage` и оставь CX'у перенести. Лучше лишний файл здесь, чем потерянный сигнал.

## 11. Пересечение с `tests/e2e/`

Каждый закрытый bug (`status: closed`) должен либо:
- Иметь активный regression-тест (ссылка в Resolution на `tests/e2e/<area>.spec.ts`), либо
- Иметь draft skeleton в `tests/e2e/pending/` (ссылка на `.draft`-файл), либо
- Иметь явную пометку «regression test declined by QA, причина: X» в Resolution.

Это гарантирует, что Rate Setter не будет исправлен, а через месяц тот же баг вернуть из git-истории.

## 12. Изменения этого документа

Формат bug-файла — шаблон, а не догма. Если ты обнаружил, что поле стало бесполезным или нужно новое — открой PR к этому README и обсуди с QA + CoS. Не меняй формат в одном файле в одностороннем порядке, иначе grep-автоматизация (будущая) сломается.
